Cloud Unit Types

Heat Reuse - Ideal for Cloud Unit owners who require continuous heat year-round. For example swimming pool or aquatic ponds.

Heat storage - Do you frequently need additional heat during the day? The buffer tank can store heat, ready to be used when extra heat is needed. Ideal for offices, residential housing, industrial processes or laundry facilities. The Cloud Unit can be utilized for heating both tap water and providing radiant floor heating.

Cooling Towers - Is your primary concern compute power and low latency? By incorporating the cooling towers, we can efficiently dissipate heat that isn't reused.

Heat capacity specifications

The default Cloud Unit configuration consists of two compute modules that contains each 2.5kW of compute power and a 1000 liter water buffer tank. Depending on the heat recipient use case we can deliver different temperatures.

When will the "hot water buffer tank" be full?

In short, it depends on several factors:

  • Heat recipient profile: can you use the heat the whole year (eg. swimming pool)?. Or do you use it mainly during the day (eg. houses and offices)?
  • Seasons and expected ambient temperature.
  • Expected outlet temperature:
    • Aquaculture (fish farming) and Food and Beverage Industry: ~30C or lower.
    • Direct underfloor heating (housing): 30C and 45C.
    • Breweries and distilleries (fermentation or mashing process): 35C - 65C.
    • Pre-heating for central heating system (+tap water): +50C
    • Textile industry: 30C to 65C
    • Paper and pulp industries: +65C
  • Required heat capacity: the Cloud Unit can generate a maximum of 120 kWh per day (5kW * 24 hours). For example if the desired outlet temperature is 70C then the storage capacity of the buffer tank will be ~70kWh (1000KG * 4180 (Joule/KG*Kelvin) * 60 Kelvin) . That means that the Cloud Unit can buffer the heat for 14 hours, if the heat is not used within that period then the excess heat will be dissipated into the air.
  • Return temperature (cold water buffer tank): Inlet temperature of the heat exchanger.
  • Heat recipient installation: Depending on your local heat installation. For example: if your installation mixes the heat exchanger outlet water then it could be effective to generate +70C water.

Share your needs and we will calculate the optimal setup for you. We continuously monitor the efficiency of the Cloud Unit and adjust the parameters to achieve the highest efficiency (season, workload, customer needs etc).

The uDC will always be operational and you can use the heat of the uDC at any time. When the hot water is not used for a while (see graph below) then the maximum capacity of the "hot water buffer tank" will be reached. At this point the heat will be wasted and dissipated into the air. You will not be liable, it just ensures that the compute customers can compute on the Cloud Unit even if the heat cannot be reused.

Software

Programming Language Suppport

In general, any binary compiled for Linux or FreeBSD is supported. Besides running binaries we also provide FaaS instances with a Software Development Kit (SDK) for the following programming languages:

  • .NET (6.0/7.0)
  • Go
  • PHP8
  • Python3
  • Rust
  • Just ask, and we can provide the required SDK FaaS environment for you.

    • Schedular

    A simple POST API request can be used to create a FaaS instance. A JSON data structure is transmitted within the request body of the API request. The JSON data contains the parameters like the payload source URL, SDK version and datastore destination.

    The task schedular ensures that FaaS instances are distributed equally between the servers.

    Deploy your code

    The code that need to run on the FaaS instance can be downloaded from anywhere. And the result can also be uploaded anywhere.

    If post-processing is needed by another FaaS instance its is recommeded to store the result on the local datastore within the Cloud Unit (uDC). The network bandwidth between the FaaS instances is 10Gbps (within the same uDC).

    Reliability

    When the connection to the local uDC is lost, you can switch to our uDC in the cloud.
